# Why Kubernetes? - Container Orchestration Fundamentals

📺 [Watch on YouTube](https://www.youtube.com/watch?v=lXs1VCWqIH4)

## Overview

Video 4 in the CKA series. Explains why Kubernetes is needed by discussing the challenges of running containers without orchestration: container crashes, off-hours support, scaling issues, deployment management, networking, security, high availability, and service discovery. Also discusses when Kubernetes is NOT the right solution.

## Key Concepts

### Challenges Without Orchestration
- Container crashes require manual intervention (SSH, check logs, restart)
- 24/7 support across time zones is expensive for manual management
- Multiple simultaneous container failures overwhelm small teams
- VM failure = entire application down (no fault tolerance)
- Deploying new versions across hundreds of containers is complex
- Exposing applications to users requires manual load balancer/routing setup
- Networking, resource management, security, HA, service discovery all need manual handling

### What Kubernetes Solves
- Automatic container restart and self-healing
- Scalability and load balancing
- Rolling deployments and rollbacks
- Service discovery and networking
- Resource management
- High availability and fault tolerance
- Orchestration across multiple nodes

### When Kubernetes is NOT Needed
- Small applications with only a few containers
- Waste of resources and money for simple workloads
- Even managed services (AKS, EKS, GKE) require administrative effort (upgrades, patching, optimization)
- Alternatives for small workloads:
  - Docker Compose
  - Containers on bare-metal/VM
  - Virtual Private Servers (DigitalOcean Droplets, AWS Lightsail)
  - Cloud marketplace images
