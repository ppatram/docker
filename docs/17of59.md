# Video 17 of 59 - Resource Requests and Limits in Kubernetes

📺 https://www.youtube.com/watch?v=Q-mk6EZVX_Q

## Overview

This video covers Resource Requests and Limits in Kubernetes — how the scheduler uses CPU and memory requests to make scheduling decisions, and how limits prevent pods from consuming all node resources. It demonstrates using the Metrics Server to observe resource utilization, stress testing pods to trigger OOM (Out of Memory) kills, and what happens when pods request more resources than available on any node.

### Installing Metrics Server

```bash
# Apply metrics server YAML
kubectl apply -f metricserver.yaml

# Verify metrics server is running
kubectl get pods -n kube-system

# Check node resource utilization
kubectl top node
```

### Creating namespace and testing memory requests/limits

```bash
# Create a namespace for the demo
kubectl create namespace mem-example

# Apply memory request pod (requests 100Mi, limits 200Mi, stress at 150Mi)
kubectl apply -f mem-request.yaml

# Check pod status
kubectl get pods -n mem-example

# Check pod resource usage
kubectl top pod memory-demo -n mem-example
```

### Testing OOM Kill (exceeding memory limit)

```bash
# Apply pod that exceeds its memory limit (requests 50Mi, limits 100Mi, stress at 250Mi)
kubectl apply -f mem2.yaml

# Check pod status - will show OOMKilled
kubectl get pods -n mem-example

# Describe pod to see OOM error
kubectl describe pod memory-demo-two -n mem-example
# Shows: Reason: OOMKilled
```

### Testing insufficient node memory

```bash
# Apply pod requesting more memory than the node has (1000Gi)
kubectl apply -f mem3.yaml

# Check pod status - will be Pending
kubectl get pods -n mem-example

# Describe pod to see insufficient memory error
kubectl describe pod <pod-name> -n mem-example
# Shows: Insufficient memory, nodes are available but none have enough resources
```

### Checking pod metrics

```bash
# Top pod for memory demo (working pod)
kubectl top pod memory-demo -n mem-example
# Shows ~150Mi memory usage, ~364m CPU

# Top pod for OOM-killed pod (no metrics since pod crashed)
kubectl top pod memory-demo-two -n mem-example
```

## YAML Manifests

### Pod with memory request and limit (within bounds)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: memory-demo
  namespace: mem-example
spec:
  containers:
    - name: memory-demo-ctr
      image: polinux/stress
      resources:
        requests:
          memory: "100Mi"
        limits:
          memory: "200Mi"
      command: ["stress"]
      args: ["--vm", "1", "--vm-bytes", "150M", "--vm-hang", "1"]
```

## Key Concepts

- **Requests**: Minimum resources guaranteed to a pod; used by the scheduler for placement decisions.
- **Limits**: Maximum resources a pod can use; exceeding memory limit triggers OOMKill.
- **CPU throttling**: Pod can't exceed CPU limit but won't be killed (just throttled).
- **OOM Kill**: When a pod exceeds its memory limit, it's killed with OOMKilled status.
- **Node OOM Killer**: If no limits are set, a pod can consume all node memory, causing the node's OOM killer to evict pods.
- **Pending state**: When requested resources exceed available node capacity, the pod stays in Pending state with "Insufficient memory" error.
- **Mi vs MB**: Mi (Mebibyte) is the unit used in Kubernetes resource specs (slightly different from MB but approximately equivalent).
