# Video 37 of 59 - Logging, Monitoring, and Debugging with crictl

📺 [Watch on YouTube](https://www.youtube.com/watch?v=cNPyajLASms)

## Overview

This video covers Kubernetes logging and monitoring basics, including installing and configuring the Metrics Server, using `kubectl top` for resource monitoring, and using `crictl` for container-level debugging. Since Kubernetes 1.24+ uses containerd instead of Docker, `crictl` replaces Docker CLI commands for troubleshooting at the container runtime level. This is essential when the API server is down and `kubectl` commands don't work.

## Key Concepts

- Kubernetes does not have built-in monitoring — requires add-ons like Metrics Server
- cAdvisor collects metrics from the container runtime and publishes to kubelet
- Kubelet forwards metrics to Metrics Server, which exposes them via the Metrics API
- From Kubernetes 1.24+, containerd is the default runtime (not Docker)
- `crictl` is the CLI client for containerd (replaces `docker` commands)
- Logs in Kubernetes are emitted to stdout/stderr and are not persisted externally by default

## Terminal Sessions

### Installing Metrics Server

```bash
kubectl get nodes
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
kubectl get pods -n kube-system
```

### Troubleshooting Metrics Server (TLS certificate issue)

```bash
kubectl describe pod <metrics-server-pod> -n kube-system
kubectl logs <metrics-server-pod> -n kube-system
kubectl get deployment -n kube-system
kubectl delete deploy metrics-server -n kube-system
```

### Re-installing and fixing Metrics Server

```bash
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
kubectl get deploy -n kube-system
kubectl describe deployment metrics-server -n kube-system
kubectl get pod -n kube-system
kubectl logs <metrics-server-pod> -n kube-system
```

### Editing Metrics Server deployment to add TLS insecure flag

```bash
kubectl edit deployment metrics-server -n kube-system
# Add under container args:
#   - --kubelet-insecure-tls
```

### Using kubectl top (after Metrics Server is ready)

```bash
kubectl top nodes
kubectl top pods
```

### Viewing pod logs

```bash
kubectl logs <pod-name>
kubectl logs <pod-name> -n kube-system
# For multi-container pods:
kubectl logs <pod-name> -c <container-name>
```

### Using crictl (containerd CLI)

```bash
# Fix permissions if needed
chmod 777 /usr/bin/crictl

# List running containers
crictl ps

# List all containers (including exited)
crictl ps -a

# Pull an image
crictl pull nginx
crictl pull redis

# List images
crictl images
```

### Simulating API server failure and debugging with crictl

```bash
cd /etc/kubernetes/manifests/
sudo mv kube-apiserver.yaml /tmp/

# kubectl will fail now
kubectl get nodes
# Error: connection to the server was refused

# Use crictl to inspect containers
crictl ps
crictl ps | grep api-server
# API server not running

# Restore API server
sudo mv /tmp/kube-apiserver.yaml /etc/kubernetes/manifests/

# Verify with crictl
crictl ps | grep api-server
# API server is back

# kubectl works again
kubectl get nodes
```

### Viewing container logs with crictl

```bash
crictl ps
# Copy container ID
sudo crictl logs <container-id>

# Tail logs
sudo crictl logs -f <container-id>
```
