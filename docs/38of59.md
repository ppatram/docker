# Video 38 of 59 - Application Failure Troubleshooting

📺 [Watch on YouTube](https://www.youtube.com/watch?v=Mil0HUtPg6I)

## Overview

This video covers application failure troubleshooting in Kubernetes using a sample multi-tier voting application (forked from Docker samples). The application has 5 microservices: a Python voting frontend, a Node.js result frontend, Redis (in-memory cache), a .NET worker, and a PostgreSQL database. The video demonstrates fixing issues with Service selectors, target ports, and NetworkPolicy rules — all common CKA exam scenarios.

## Architecture

- **Vote** (Python frontend) → exposed on NodePort 31000
- **Redis** (in-memory cache) → ClusterIP service, collects votes
- **Worker** (.NET) → consumes votes from Redis, stores in PostgreSQL
- **PostgreSQL** (database) → ClusterIP service, stores votes
- **Result** (Node.js frontend) → exposed on NodePort 31001, reads from DB

## Terminal Sessions

### Deploying the sample application

```bash
kubectl get nodes
kubectl get all
cd k8s-specifications/
kubectl apply -f .
```

### Verifying deployments, pods, and services

```bash
kubectl get deploy
kubectl get pods
kubectl get services
kubectl get netpol
kubectl get pods -o wide
kubectl get nodes -o wide
```

### Troubleshooting Vote service (selector mismatch)

```bash
kubectl describe svc vote
# Endpoints: <none> -- indicates selector mismatch

kubectl get svc vote -o yaml
# selector: app: votes (incorrect - has extra 's')

kubectl get pods --show-labels
# Pod label: app=vote

kubectl edit svc vote
# Change selector from "app: votes" to "app: vote"

kubectl describe svc vote
# Endpoints now populated

kubectl get endpoints
```

### Troubleshooting Result service (wrong target port)

```bash
kubectl describe svc result
# Endpoints exist, TargetPort: 8080

kubectl get pod <result-pod> -o yaml
# containerPort: 80 (mismatch with targetPort 8080)

kubectl edit svc result
# Change targetPort from 8080 to 80

kubectl describe svc result
# TargetPort now 80
```

### Troubleshooting Redis connectivity (NetworkPolicy issue)

```bash
kubectl get netpol
kubectl describe netpol <redis-network-policy>
# Ingress from pod selector: app=frontend
# But vote pod label is: app=vote

kubectl get pods --show-labels
# Vote pod has label: app=vote (not app=frontend)

kubectl edit netpol <redis-network-policy>
# Change from matchLabels app: frontend to app: vote

kubectl describe netpol <redis-network-policy>
# Now allows traffic from pods with app=vote
```

### Checking worker and database pods

```bash
kubectl get all
kubectl get pods
kubectl get svc
kubectl describe svc db
kubectl edit deploy db
kubectl edit deploy worker
```

### Checking pod logs

```bash
kubectl logs <result-pod>
# "relation votes does not exist"

kubectl delete pod <result-pod>
# Deployment recreates it

kubectl get deploy
kubectl logs <worker-pod>
```

### Restarting pods

```bash
kubectl delete pod result-<hash>
kubectl delete pod worker-<hash>
kubectl get pods
```
