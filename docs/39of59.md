# Video 39 of 59 - Control Plane Failure Troubleshooting

📺 [Watch on YouTube](https://www.youtube.com/watch?v=z6XjbuRl6LE)

## Overview

This video covers troubleshooting control plane failures in Kubernetes — a key CKA exam topic. It walks through scenarios involving a broken API server (typo in command), a broken kube-scheduler (wrong image tag), and a broken kube-controller-manager (typo in command name and incorrect volume path). Each scenario demonstrates how to diagnose and fix issues using crictl, kubectl logs, kubectl describe, and editing static pod manifests.

## Key Concepts

- Control plane components run as static pods with manifests in `/etc/kubernetes/manifests/`
- When API server is down, use `crictl` for container-level debugging
- kube-scheduler issues cause pods to remain in Pending state
- kube-controller-manager issues prevent scaling and desired state reconciliation
- Always verify image tags, command names, and volume mount paths in manifests

## Terminal Sessions

### Scenario 1: Broken API Server

```bash
# kubectl fails
kubectl get nodes
# Error: connection to the server was refused

# Check if API server container is running
crictl ps | grep api
# Not found

# Check exited containers
crictl ps -a | grep api
# Shows exited API server container

# Check logs of exited container
crictl logs <container-id>

# Open API server manifest and fix typo
sudo vi /etc/kubernetes/manifests/kube-apiserver.yaml
# Fix: "kube-apiserverr" -> "kube-apiserver" (remove extra 'r' in command)

# Verify API server comes back
crictl ps | grep api
kubectl get nodes
```

### Fixing kubeconfig issues

```bash
# If KUBECONFIG env var is wrong
export KUBECONFIG=/tmp/config  # wrong path

kubectl get nodes
# Error: connection to localhost:8080 refused

# API server is actually running
crictl ps | grep api

# Fix by using correct kubeconfig
kubectl get nodes --kubeconfig /etc/kubernetes/admin.conf
sudo chmod 775 /etc/kubernetes/admin.conf

# Or set correct KUBECONFIG
export KUBECONFIG=$HOME/.kube/config
kubectl get nodes
```

### Scenario 2: Broken kube-scheduler (wrong image)

```bash
# Pod stuck in Pending
kubectl run nginx --image=nginx
kubectl get pods
# Status: Pending

kubectl describe pod nginx
# Node: <none> -- not scheduled

# Check scheduler pod
kubectl get pods -n kube-system
# kube-scheduler-master: ImagePullBackOff

kubectl logs kube-scheduler-master -n kube-system
# Failed to pull image

kubectl describe pod kube-scheduler-master -n kube-system
# Image: registry.k8s.io/kube-scheduler:v1.30.200x (wrong tag)

# Fix scheduler manifest
sudo vi /etc/kubernetes/manifests/kube-scheduler.yaml
# Compare with API server to get correct tag:
sudo vi /etc/kubernetes/manifests/kube-apiserver.yaml
# Image tag should be: v1.30.2

# Fix the image tag in kube-scheduler.yaml
# Change: v1.30.200x -> v1.30.2

# Wait for scheduler to come up
kubectl get pods -n kube-system -w
kubectl get pods
# nginx pod now Running
```

### Scenario 3: Broken kube-controller-manager (typo in command)

```bash
kubectl get deploy
# 2 replicas

kubectl delete pod <nginx-pod>
# New pod is NOT recreated

kubectl get pods -n kube-system
# kube-controller-manager-master: CrashLoopBackOff

kubectl describe pod kube-controller-manager-master -n kube-system
# Error: executable file "kube-controller-managere" not found

# Fix the command typo
sudo vi /etc/kubernetes/manifests/kube-controller-manager.yaml
# Fix: "kube-controller-managere" -> "kube-controller-manager"

# Wait for controller manager to restart
kubectl get pods -n kube-system -w
kubectl get pods
# Deployment now maintains desired replica count
```
