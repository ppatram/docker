# Video 16 of 59 - Node Affinity in Kubernetes

📺 https://www.youtube.com/watch?v=5vimzBRnoDk

## Overview

This video covers Node Affinity in Kubernetes — a scheduling concept that allows pods to be scheduled on specific nodes based on label expressions and operators. It explains the difference between `requiredDuringSchedulingIgnoredDuringExecution` and `preferredDuringSchedulingIgnoredDuringExecution`, demonstrates various operators (In, Exists, NotIn), and compares Node Affinity with Taints and Tolerations showing how to use both together for precise pod placement.

### Applying Affinity with requiredDuringScheduling (no matching label)

```bash
# Apply the affinity manifest
kubectl apply -f affinity.yaml

# Check pod status - will be Pending since no matching label exists
kubectl get pods

# Describe pod to see scheduling failure reason
kubectl describe pod <pod-name>
# Output: "0 of 3 nodes are available... two nodes did not match pod affinity/selector"
```

### Adding a label to the node to satisfy affinity

```bash
# Check current node labels
kubectl get nodes --show-labels

# Add the disk-type=ssd label to a worker node
kubectl label node <node-name> disk-type=ssd

# Verify the pod is now running
kubectl get pods

# Check which node the pod was scheduled on
kubectl get pods -o wide
# Shows the pod running on the labeled node
```

### Creating preferred affinity with non-matching label

```bash
# Create a second affinity file
vim affinity2.yaml

# Apply the preferred affinity manifest (with a non-existing label value "sdd")
kubectl apply -f affinity2.yaml

# Check pods - pod is scheduled even though label doesn't match
kubectl get pods
# Pod is Running because preferredDuringScheduling allows scheduling on any node
```

### Using the Exists operator

```bash
# Remove the label from the node
kubectl label node <node-name> disk-type-

# Re-label with blank value
kubectl label node <node-name> disk-type=

# Verify labels
kubectl get nodes --show-labels
# Shows disk-type with no value

# Apply manifest using Exists operator (only checks if label key exists)
kubectl apply -f affinity.yaml

# Check pods - scheduled because the label key exists (value doesn't matter)
kubectl get pods
kubectl get pods -o wide
```

## YAML Manifests

### Node Affinity with requiredDuringSchedulingIgnoredDuringExecution

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: redis
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: disk-type
                operator: In
                values:
                  - ssd
  containers:
    - name: redis
      image: redis
```

### Node Affinity with preferredDuringSchedulingIgnoredDuringExecution

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: redis-new
spec:
  affinity:
    nodeAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 1
          preference:
            matchExpressions:
              - key: disk-type
                operator: In
                values:
                  - sdd
  containers:
    - name: redis
      image: redis
```

### Node Affinity with Exists operator

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: redis-three
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: disk-type
                operator: Exists
  containers:
    - name: redis
      image: redis
```

## Key Concepts

- **requiredDuringSchedulingIgnoredDuringExecution**: Pod ONLY gets scheduled if a matching label exists on a node. If no match, pod stays Pending.
- **preferredDuringSchedulingIgnoredDuringExecution**: Pod PREFERS nodes with matching labels, but will schedule on any available node if no match.
- **IgnoredDuringExecution** (both): Changes to node labels after pod is scheduled do NOT affect running pods.
- **Operators**: `In`, `NotIn`, `Exists`, `DoesNotExist`, `Gt`, `Lt`
- **Best Practice**: Use Node Affinity + Taints/Tolerations together for precise scheduling control.
