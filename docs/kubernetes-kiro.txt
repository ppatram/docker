  ───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────

  What's Fast to Create/Destroy

  │ **VPC Endpoints**                                                                     │ 2–5 min       │ 2–3 min                   │ Mildly slow             │

  └───────────────────────────────────────────────────────────────────────────────────────┴───────────────┴───────────────────────────┴─────────────────────────┘

  Total cold-start time: ~40–60 minutes before you can deploy your first pod. Then another 40+ minutes to tear down cleanly.

  ───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────

  The Real Friction

  Beyond raw time, these issues make full create/destroy impractical for regular training:

  1. EKS add-ons and Helm charts (ALB controller, external-secrets, metrics-server) need reinstalling every time
  2. IAM OIDC provider must be re-associated and service accounts re-linked
  3. DNS/ACM validation — if you're using real domains, certificate validation takes extra time
  4. Ordering dependencies on destroy — you can't delete a VPC until all ENIs, NAT GWs, and endpoints are gone. Terraform/CDK handles this but sometimes hangs on stuck resources
  5. State drift — if a destroy fails halfway, manual cleanup is painful

  ───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────

  Practical Middle Ground

  Keep the slow/cheap stuff running. Destroy the expensive stuff.

  ┌───────────────────────────────┬─────────────────────────────────────────────────────┐
  │ Keep Running (~$3–5/day)      │ Create/Destroy Per Session                          │
  ├───────────────────────────────┼─────────────────────────────────────────────────────┤
  │ VPC, subnets, route tables    │ EKS node group (scale to 0 when idle)               │
  │ Security groups               │ RDS (or stop/start it)                              │
  │ IAM roles, OIDC provider      │ ElastiCache                                         │
  │ ECR repository                │ NAT Gateway ($1.08/day — destroy if cost-sensitive) │
  │ Secrets Manager secrets       │ ALB (only exists when Ingress is deployed)          │
  │ EKS control plane ($2.40/day) │                                                     │
  └───────────────────────────────┴─────────────────────────────────────────────────────┘

  Cost while idle: ~$5–7/day ($150–210/month) with just VPC + EKS control plane running.

  Best trick: Scale the node group to 0 when not training:

  aws eks update-nodegroup-config \
    --cluster-name training \
    --nodegroup-name workers \
    --scaling-config minSize=0,maxSize=2,desiredSize=0 \
    --region us-east-1

  This drops EC2 costs to $0 while keeping the cluster ready. Scale back up in ~3 minutes when you need it.

  ───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────

  Recommendation

  If you're training weekly or more, keep VPC + EKS running and scale nodes to 0 between sessions. If it's monthly or less, full teardown with Terraform/CDK is reasonable — just budget an hour of setup time
  before each session.

  terraform init

  terraform apply

  After apply, configure kubectl:

  $(terraform output -raw kubeconfig_command)

  To pause (scale nodes to 0, save ~$4/day on EC2):

  $(terraform output -raw scale_down_command)

  To resume:

  $(terraform output -raw scale_up_command)

  Key Design Decisions

  - Node min_size = 0 — allows scaling to zero between training sessions
  - ElastiCache disabled by default — set enable_elasticache = true when ready to teach caching
  - NAT Gateway toggleable — can be disabled if you only need intra-VPC traffic
  - Single NAT Gateway (not per-AZ) — saves ~$65/month vs production multi-AZ pattern
  - RDS skip_final_snapshot = true — appropriate for training, not production
  - Outputs include scale commands — easy copy-paste for pause/resume

     1+  # EKS Training Infrastructure — Architecture Diagram
     2+
     3+  ## Overview
     4+
     5+  This diagram shows the AWS infrastructure provisioned by the Terraform root module for Kubernetes training. It follows a single-region, multi-AZ pattern optimized for cost.
     6+
     7+  ## Request Flow
     8+
     9+  1. **User** → Internet Gateway → ALB (TLS terminated via ACM certificate)
    10+  2. **ALB** → EKS Worker Nodes (port 8080) in private subnets
    11+  3. **Worker Nodes** run API pods across namespaces (api-dev, api-qa, api-prod)
    12+  4. **API Pods** → RDS PostgreSQL (port 5432) and optionally ElastiCache Redis (port 6379) in data subnets
    13+  5. **Worker Nodes** → NAT Gateway → Internet Gateway for outbound traffic (ECR pulls, external APIs)
    14+
    15+  ## Components
    16+
    17+  | Service | Purpose |
    18+  |---|---|
    19+  | **VPC** | Network isolation with public, private, and data subnets |
    20+  | **Internet Gateway** | Inbound internet access to public subnets |
    21+  | **NAT Gateway** | Outbound internet for private subnets (single AZ for cost) |
    22+  | **ALB** | HTTPS ingress with TLS termination, path-based routing |
    23+  | **ACM** | TLS certificates for ALB |
    24+  | **EKS Cluster** | Kubernetes control plane |
    25+  | **Worker Nodes** | EC2 t3.medium instances running pods |
    26+  | **HPA / Metrics Server** | Horizontal Pod Autoscaler for production overlay |
    27+  | **RDS PostgreSQL** | Application database (db.t3.micro, encrypted) |
    28+  | **ElastiCache Redis** | Optional caching layer (disabled by default) |
    29+  | **ECR** | Container image registry with scan-on-push |
    30+  | **Secrets Manager** | DB credentials injected via External Secrets Operator |
    31+  | **KMS** | Encryption at rest for RDS and ElastiCache |
    32+  | **CloudWatch** | Centralized logs, metrics, and Container Insights |
    33+  | **SNS** | Alert delivery from CloudWatch Alarms |
    34+  | **X-Ray** | Distributed request tracing |
    35+
    36+  ## Security Controls
    37+
    38+  - TLS 1.2+ on ALB ingress (ACM certificate)
    39+  - Worker nodes in private subnets (no public IPs)
    40+  - Security groups per tier: ALB → Nodes → Data
    41+  - KMS encryption at rest for RDS and ElastiCache
    42+  - Secrets Manager for credentials (never in code or env files)
    43+  - ECR image scanning on push
    44+  - Network policies restrict pod-to-pod traffic
    45+
    46+  ## Cost Optimization
    47+
    48+  - NAT Gateway: single AZ (not per-AZ)
    49+  - ElastiCache: disabled by default
    50+  - Node group: min_size=0 allows scaling to zero between sessions

  does an aws api gateway make this better?

  For a Kubernetes training environment, no — API Gateway would add cost and complexity without much benefit. Here's the comparison:

  ALB (current setup) vs API Gateway + ALB

  ┌────────────────────────┬────────────────────────────────────────────────┬────────────────────────────────────────────────────────────────────────┐
  │ Factor                 │ ALB Only (current)                             │ Adding API Gateway                                                     │
  ├────────────────────────┼────────────────────────────────────────────────┼────────────────────────────────────────────────────────────────────────┤
  │ **Cost**               │ ~$22/month + LCUs                              │ +$3.50/million requests + $22 ALB still needed                         │
  │ **Complexity**         │ Simple — Ingress resource creates ALB directly │ Extra hop: API GW → VPC Link → NLB/ALB → pods                          │
  │ **Latency**            │ Single hop to pods                             │ Adds 5-15ms per request                                                │
  │ **What you gain**      │ —                                              │ Throttling, API keys, request validation, usage plans, WAF integration │
  │ **K8s learning value** │ Teaches Ingress, ALB controller, services      │ Teaches... API Gateway (not a K8s concept)                             │
  └────────────────────────┴────────────────────────────────────────────────┴────────────────────────────────────────────────────────────────────────┘

  When API Gateway *would* make sense

  - Production APIs needing rate limiting, API keys, and usage plans per consumer
  - Request transformation (mapping templates between clients and backend)
  - Multiple backends where some are Lambda and some are EKS
  - Fine-grained authorization (Lambda authorizers, Cognito)
  - WAF without paying for ALB WAF separately

  For your training purposes

  Stick with ALB. The whole point is learning Kubernetes networking — Ingress resources, Services, ALB controller annotations, path-based routing. API Gateway would obscure those concepts behind a managed
  proxy that has nothing to do with Kubernetes.

  If you want to teach API management later, you can add it without changing the EKS setup (API GW → VPC Link → existing ALB or NLB).

  would this terraform apply build using cloudformation or jsut api calls to aws

  Just direct API calls to AWS. Terraform does not use CloudFormation.

  When you run terraform apply, the AWS provider makes direct REST API calls to each service (EC2 API for VPCs/subnets, EKS API for clusters, RDS API for databases, etc.). It tracks the state of what it
  created in a local terraform.tfstate file (or remote backend like S3).

  The one exception in this stack: EKS itself uses CloudFormation internally on the AWS side to orchestrate node group creation. You'll see a CloudFormation stack appear in the console named something like
  eks-... — but that's AWS's internal implementation detail, not something Terraform triggers directly. Terraform just calls the EKS API and waits for it to complete.

  Comparison

  ┌────────────────────┬───────────────────────────────────────────────────────────────────────┐
  │ Tool               │ How it provisions                                                     │
  ├────────────────────┼───────────────────────────────────────────────────────────────────────┤
  │ **Terraform**      │ Direct AWS API calls, state in `.tfstate`                             │
  │ **AWS CDK**        │ Generates CloudFormation templates, deploys via CloudFormation stacks │
  │ **CloudFormation** │ Declarative templates deployed as stacks                              │
  │ **AWS SAM**        │ CloudFormation extension for serverless                               │
  └────────────────────┴───────────────────────────────────────────────────────────────────────┘

  This is why Terraform destroy is generally faster and more reliable for cleanup — it deletes resources directly via API rather than going through CloudFormation stack deletion ordering.

  will pandoc convert drawio to png

  No. Pandoc doesn't support .drawio files. It handles document format conversions (markdown, HTML, PDF, docx), not diagram rendering.

  To convert .drawio to PNG, use the draw.io CLI (also called drawio or xdg-open):

  # Linux (install draw.io desktop or use the CLI package)
  drawio -x -f png -o output.png input.drawio

  # macOS
  /Applications/draw.io.app/Contents/MacOS/draw.io -x -f png -o output.png input.drawio

  Or alternatives:

  ┌────────────────────┬─────────────────────────────────────────────────────────────────────┐
  │ Method             │ Command/Tool                                                        │

  why is the xvfb-run using a gpu?

  It's not actually using a GPU — that error message is harmless.

  Draw.io desktop is an Electron app (Chromium-based). Chromium always tries to initialize a GPU process for hardware-accelerated rendering. On this headless server there's no GPU, so the GPU process fails and
  Chromium falls back to software rendering automatically.

  The log line:

  Exiting GPU process due to errors during initialization

  ...just means "no GPU found, falling back to CPU rendering." The export still works fine.

  You can suppress the warning with --disable-gpu:

  xvfb-run /tmp/drawio-app/opt/drawio/drawio --disable-gpu -x -f png -o output.png input.drawio

  But it's purely cosmetic — the output is identical either way.

  can I run gitlab in kubernetes?

  Yes, GitLab publishes an official Helm chart for running on Kubernetes. But for a training environment, I'd advise against it:

  Why it's impractical here:

  ┌───────────────────────┬────────────────────────────────────────────────────────────────────────────────────────────┐
  │ Concern               │ Impact                                                                                     │
  ├───────────────────────┼────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Resource requirements │ GitLab needs 8+ GB RAM minimum (webservice, Sidekiq, Gitaly, PostgreSQL, Redis, registry)  │
  │ Your nodes            │ 2× t3.medium = 8 GB total — GitLab alone would consume your entire cluster                 │
  │ Complexity            │ 30+ pods, persistent volumes, certificates, DNS — more infra work than the actual training │
  │ Cost                  │ You'd need to upsize to t3.xlarge or add nodes (~$100+/month extra)                        │
  └───────────────────────┴────────────────────────────────────────────────────────────────────────────────────────────┘

  Better alternatives for training:

  1. GitLab.com free tier — 5 GB storage, CI/CD included, zero maintenance
  2. Gitea on K8s — lightweight Git server (~256 MB RAM), good if you want to practice deploying a stateful app
  3. Local git + manual `kubectl apply` — simplest for learning Kubernetes itself

  If your goal is to learn Kubernetes concepts (deployments, services, ingress, kustomize), use GitLab.com for source control and focus your cluster resources on the actual application workloads.

  If your goal is specifically to learn how to run stateful apps on Kubernetes, then Gitea or a minimal GitLab Omnibus single-pod deployment would be a good exercise — just budget a larger node.

  how do I build and deploy that docker image?

  # 1. Get your ECR repository URL
  ECR_URL=$(cd terraform && terraform output -raw ecr_repository_url)

  # 2. Login to ECR
  aws ecr get-login-password --region us-east-1 | docker login --username AWS --password-stdin "$ECR_URL"

  # 3. Build the image
  docker build -t "$ECR_URL:v1" app/

  # 4. Push to ECR
  docker push "$ECR_URL:v1"

  # 5. Update the dev overlay to use your image
  cd manifests/overlays/dev
  kustomize edit set image "api=$ECR_URL:v1"

  # 6. Deploy to dev namespace
  kubectl apply -k .

  # 7. Verify
  kubectl get pods -n api-dev
  kubectl rollout status deployment/api -n api-dev

  Test the deployed app:

  # Port-forward to test without ALB
  kubectl port-forward -n api-dev svc/api 8080:80 &

  curl -X POST http://localhost:8080/api/v1/hello \
    -H "Content-Type: application/json" \
    -d '{"name":"kubernetes"}'

  If the ALB Ingress is set up, you can get the external URL with:

  kubectl get ingress -n api-dev -o jsonpath='{.items[0].status.loadBalancer.ingress[0].hostname}'
