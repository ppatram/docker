# Video 12 of 59 - Multi-Container Pods, Init Containers, and Environment Variables

https://www.youtube.com/watch?v=yRiFq1ykBxc

## Overview

This video covers multi-container pods in Kubernetes, including init containers and sidecar containers. It demonstrates how init containers run before the main app container and block its startup until they complete. It also covers environment variables in pods and how to use commands and arguments in container specs.

## Key Concepts

- Init containers run before app containers and must complete before the app starts
- Sidecar containers run alongside the app container for the pod's lifetime
- All containers in a pod share the same resources (CPU, memory, storage)
- Environment variables can be set in pod specs using `env` field
- Commands and arguments can be passed to containers using `command` and `args` fields

## Terminal Sessions

### Creating a basic pod with environment variables

```bash
kubectl create -f pod.yaml
kubectl get pod
kubectl delete pod my-app
```

### Pod YAML with init container

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  labels:
    app: my-app-pod
spec:
  containers:
  - name: my-app-container
    image: busybox
    env:
    - name: FIRST_NAME
      value: "Piyush"
    command: ['sh', '-c', 'echo the app is running && sleep 3600']
  initContainers:
  - name: init-my-service
    image: busybox:1.28
    command: ['sh', '-c']
    args: ['until nslookup my-service.default.svc.cluster.local; do echo waiting for service to be up; sleep 2; done']
```

### Applying and monitoring the init container

```bash
kubectl create -f pod.yaml
kubectl get pod
# Shows: init 0/1 (waiting for init container to complete)

kubectl describe pod my-app
kubectl logs my-app -c init-my-service
```

### Creating a deployment and service to satisfy init container

```bash
# Create nginx deployment
kubectl create deploy nginx-deploy --image=nginx --port=80
kubectl get deploy

# Expose service with name matching what init container expects
kubectl expose deploy nginx-deploy --name my-service --port 80
kubectl get pods
# Pod transitions from Init to Running

# Check init container logs after completion
kubectl logs my-app -c init-my-service
```

### Exec into pod and check environment variables

```bash
# Run inline command
kubectl exec -it my-app -- printenv

# Open shell in container
kubectl exec -it my-app -- sh
echo $FIRST_NAME
```

### Adding multiple init containers

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  labels:
    app: my-app-pod
spec:
  containers:
  - name: my-app-container
    image: busybox
    env:
    - name: FIRST_NAME
      value: "Piyush"
    command: ['sh', '-c', 'echo the app is running && sleep 3600']
  initContainers:
  - name: init-my-service
    image: busybox:1.28
    command: ['sh', '-c']
    args: ['until nslookup my-service.default.svc.cluster.local; do echo waiting for service to be up; sleep 2; done']
  - name: init-my-db
    image: busybox:1.28
    command: ['sh', '-c']
    args: ['until nslookup my-db.default.svc.cluster.local; do echo waiting for service to be up; sleep 2; done']
```

### Recreating pod with multiple init containers

```bash
# Cannot add/remove containers from running pod
kubectl delete pod my-app
kubectl apply -f pod.yaml
kubectl get pod
# Shows: init 1/2 (first init done, second waiting)

# Create second deployment and service
kubectl create deploy my-db --image=redis
kubectl expose deploy my-db --name my-db --port 80

kubectl get svc
kubectl get pod
# Pod transitions to Running once both init containers complete
```
