# Video 14 of 59 - Static Pods, Node Selectors, Labels and Selectors, and Manual Scheduling

https://www.youtube.com/watch?v=6eGf7_VSbrQ

## Overview

This video covers how the Kubernetes scheduler works, static pods, manual scheduling using `nodeName`, and labels/selectors for filtering resources. Static pods are managed directly by kubelet (not the scheduler) and their manifests live in `/etc/kubernetes/manifests`. Manual scheduling bypasses the scheduler by specifying `nodeName` in the pod spec. Labels and selectors enable filtering and grouping of Kubernetes resources.

## Key Concepts

- Scheduler monitors pending pods and assigns them to nodes based on available resources
- Static pods are control plane components managed by kubelet from manifest files
- Static pod manifests are stored in `/etc/kubernetes/manifests/`
- Removing a manifest file stops the static pod; restoring it restarts the pod
- Manual scheduling uses `nodeName` field to bypass the scheduler
- Labels are key-value pairs attached to resources for identification
- Selectors filter resources by their labels
- Annotations store metadata about objects (not used for filtering)

## Terminal Sessions

### Viewing the scheduler pod

```bash
kubectl get pods --namespace kube-system | grep scheduler
kubectl get pods -n kube-system -o wide | grep scheduler
```

### Entering the control plane node (Kind cluster)

```bash
# Find the container name
docker ps | grep control-plane

# Exec into control plane node
docker exec -it cka-cluster-3-control-plane bash
```

### Exploring static pod manifests

```bash
# Inside the control plane node
ps -ef | grep kubelet

# Static pod manifests directory
ls /etc/kubernetes/manifests/
# Shows: etcd.yaml  kube-apiserver.yaml  kube-controller-manager.yaml  kube-scheduler.yaml
```

### Demonstrating static pod behavior (removing scheduler)

```bash
# Move scheduler manifest to temp (stops the scheduler)
mv /etc/kubernetes/manifests/kube-scheduler.yaml /tmp/

# Verify scheduler is no longer running (from another terminal)
kubectl get pods -n kube-system | grep scheduler
# No results - scheduler is down

# Try creating a pod - it stays pending without scheduler
kubectl run nginx --image=nginx
kubectl get pods
# Shows: nginx - Pending

kubectl describe pod nginx
# Node field is blank, status is Pending

# Restore the scheduler manifest
mv /tmp/kube-scheduler.yaml /etc/kubernetes/manifests/

# Verify scheduler is running again
kubectl get pods -n kube-system | grep scheduler
# Now running

# Pod gets scheduled
kubectl get pods
# nginx is now Running
```

### Manual scheduling with nodeName

```bash
# Generate pod YAML
kubectl run nginx --image=nginx -o yaml --dry-run=client > pod.yaml

# Delete existing pod
kubectl delete pod nginx
```

### Pod YAML with nodeName (manual scheduling)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
  labels:
    run: nginx
spec:
  nodeName: cka-cluster-3-worker
  containers:
  - name: nginx
    image: nginx
```

```bash
# Stop the scheduler again (from inside control plane node)
mv /etc/kubernetes/manifests/kube-scheduler.yaml /tmp/

# Verify scheduler is down
kubectl get pods -n kube-system | grep scheduler

# Apply pod with nodeName - works even without scheduler
kubectl apply -f pod.yaml
kubectl get pods
# Running!

kubectl get pods -o wide
# Scheduled on cka-cluster-3-worker as specified

# Restore scheduler
mv /tmp/kube-scheduler.yaml /etc/kubernetes/manifests/
kubectl get pods -n kube-system | grep scheduler
# Running again
```

### Working with labels and selectors

```bash
kubectl get pods
kubectl get pods --show-labels
```

### Pod YAML with multiple labels

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
  labels:
    run: nginx
    tier: frontend
    type: app1
spec:
  containers:
  - name: nginx
    image: nginx
```

```bash
kubectl apply -f pod.yaml --force
kubectl get pods --show-labels
```

### Creating a second pod with different labels

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: redis
  labels:
    run: redis
    tier: backend
spec:
  containers:
  - name: redis
    image: redis
```

```bash
kubectl apply -f pod2.yaml
kubectl get pods
```

### Filtering pods by labels using selectors

```bash
# Filter by tier=frontend
kubectl get pods --selector tier=frontend

# Filter by tier=backend
kubectl get pods --selector tier=backend
```

### Viewing annotations

```bash
kubectl edit pod nginx
# Shows annotations with last-applied-configuration, timestamps, etc.
```
