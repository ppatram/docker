# Video 30 of 59 - Kubernetes Storage: Persistent Volumes, PVCs, and Storage Classes

📺 https://www.youtube.com/watch?v=2NzYX8_lX_0

## Overview

This video covers Kubernetes storage concepts: emptyDir volumes (temporary), Persistent Volumes (PV), Persistent Volume Claims (PVC), access modes, reclaim policies, and Storage Classes. It demonstrates how data is lost with emptyDir when a pod is deleted, then shows how to use PV/PVC with hostPath for persistent storage. Finally, it introduces StorageClass for dynamic provisioning with cloud storage backends.

## Terminal Sessions

### Creating a Redis Pod with emptyDir Volume

```bash
vi redis.yaml
kubectl apply -f redis.yaml

kubectl get pods
```

### Redis Pod YAML (emptyDir - Non-Persistent)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: redis-pod
spec:
  containers:
    - name: redis
      image: redis
      volumeMounts:
        - name: redis-storage
          mountPath: /data/redis
  volumes:
    - name: redis-storage
      emptyDir: {}
```

### Testing emptyDir Volume (Data Survives Container Restart)

```bash
# Exec into the redis pod
kubectl exec -it redis-pod -- sh

# Navigate to the mounted volume
cd /data/redis

# Create a test file
echo "hello kubernetes" > test.txt
cat test.txt

# Install ps utility
apt-get update && apt-get install procps -y

# Kill the redis process to simulate container restart
ps aux
kill -9 1

# After container restarts, exec back in
kubectl exec -it redis-pod -- sh
cd /data/redis
ls
# test.txt is still there - survived container restart
exit
```

### Testing emptyDir Volume (Data Lost on Pod Deletion)

```bash
# Delete the pod
kubectl delete pod redis-pod

# Recreate it
kubectl apply -f redis.yaml

# Exec into the new pod
kubectl exec -it redis-pod -- sh
cd /data/redis
ls
# File is GONE - emptyDir is not persistent across pod deletions
exit
```

### Creating a Persistent Volume (PV)

```bash
kubectl apply -f pv.yaml
kubectl get pv
```

### Persistent Volume YAML (hostPath)

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: task-pv-volume
  labels:
    type: local
spec:
  capacity:
    storage: 1Gi
  accessModes:
    - ReadWriteOnce
  hostPath:
    path: "/home/ubuntu/day29"
```

### Creating a Persistent Volume Claim (PVC)

```bash
kubectl apply -f pvc.yaml
kubectl get pvc
kubectl get pv
# Status should show "Bound"
```

### Persistent Volume Claim YAML

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: task-pv-claim
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 500Mi
```

### Creating a Pod with PVC

```bash
kubectl apply -f pod.yaml
kubectl get pods
```

### Pod YAML (Using PVC)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: task-pv-pod
spec:
  nodeName: master
  containers:
    - name: nginx
      image: nginx
      volumeMounts:
        - name: task-pv-storage
          mountPath: /usr/share/nginx/html
  volumes:
    - name: task-pv-storage
      persistentVolumeClaim:
        claimName: task-pv-claim
```

### Testing PV/PVC (Verify hostPath Mounting)

```bash
# Exec into the pod
kubectl exec -it task-pv-pod -- bash

ls
# Shows files from the host path /home/ubuntu/day29

# Test nginx serves the mounted content

# Alternative: exec without entering shell

```

## Key Concepts

### Access Modes
- **ReadWriteOnce (RWO)** - Volume can be mounted as read-write by a single node
- **ReadOnlyMany (ROX)** - Volume can be mounted as read-only by many nodes
- **ReadWriteMany (RWX)** - Volume can be mounted as read-write by many nodes
- **ReadWriteOncePod (RWOP)** - Volume can be mounted as read-write by a single pod

### Reclaim Policies
- **Retain** - PV remains after PVC deletion, but is in "Released" state (cannot be claimed again)
- **Delete** - PV is deleted when PVC is deleted
- **Recycle** - PV data is scrubbed and made available again for new claims

### StorageClass Example

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: standard
provisioner: kubernetes.io/gce-pd
parameters:
  type: pd-standard
```

### Storage Class Provisioners
- `kubernetes.io/azure-file` - Azure File
- `kubernetes.io/gce-pd` - GCE Persistent Disk
- `kubernetes.io/aws-ebs` - AWS EBS
- `kubernetes.io/no-provisioner` - Local storage
- NFS, Portworx, vSphere, and many more

### Dynamic Provisioning
With dynamic provisioning, you only need to create a PVC with a `storageClassName` — the PV is automatically created by the provisioner:

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: standard
  resources:
    requests:
      storage: 10Gi
```
