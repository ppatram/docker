# Video 36 of 59 - etcd Backup and Restore

📺 [Watch on YouTube](https://www.youtube.com/watch?v=R2wuFCYgnm4)

## Overview

This video covers etcd backup and restore — a critical task for Kubernetes administrators and a common CKA exam topic. It demonstrates how to take a snapshot backup of the etcd datastore and how to restore the cluster from that backup. The video explains the etcd data directory, required certificates, and the full restore process including restarting control plane components.

## Key Concepts

- etcd stores all cluster state, configuration data, and manifest objects as key-value pairs
- Backups are taken before cluster upgrades or major releases
- The etcd data directory (default: `/var/lib/etcd`) contains all cluster data
- etcd listens on port 2379 for client connections
- Certificates (CA cert, server cert, server key) are required for etcdctl commands
- From etcd 3.6+, `etcdutl` replaces the deprecated `etcdctl` for snapshot operations

## Terminal Sessions

### Viewing cluster objects and exporting to YAML backup

```bash
kubectl get all
kubectl get all -A
kubectl get all -A -o yaml > backup.yaml
```

### Examining etcd static pod manifest

```bash
sudo cat /etc/kubernetes/manifests/etcd.yaml
```

### Installing etcdctl client

```bash
sudo apt install etcd-client
```

### Setting etcdctl API version

```bash
export ETCDCTL_API=3
```

### Taking an etcd snapshot backup

```bash
sudo ETCDCTL_API=3 etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  snapshot save /opt/etcd-backup.db
```

### Verifying the snapshot

```bash
sudo ls /opt/etcd-backup.db
sudo du -sh /opt/etcd-backup.db
```

### Checking snapshot status

```bash
sudo ETCDCTL_API=3 etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  --write-out=table \
  snapshot status /opt/etcd-backup.db
```

### Simulating data loss (deleting resources)

```bash
kubectl get all
kubectl delete deploy nginx
kubectl delete svc nginx
kubectl get all
```

### Restoring from snapshot

```bash
sudo ETCDCTL_API=3 etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  snapshot restore /opt/etcd-backup.db \
  --data-dir=/var/lib/etcd-restore-from-backup
```

### Updating etcd manifest to use restored data directory

```bash
sudo vi /etc/kubernetes/manifests/etcd.yaml
# Change the following:
#   --data-dir=/var/lib/etcd  -->  --data-dir=/var/lib/etcd-restore-from-backup
#   volumeMounts mountPath: /var/lib/etcd  -->  /var/lib/etcd-restore-from-backup
#   volumes hostPath path: /var/lib/etcd  -->  /var/lib/etcd-restore-from-backup
```

### Restarting control plane components (move manifests out and back)

```bash
cd /etc/kubernetes/manifests/
sudo mv *.yaml /tmp/
# Wait for components to stop
sudo mv /tmp/*.yaml /etc/kubernetes/manifests/
# Wait for components to restart
```

### Restarting kubelet

```bash
sudo systemctl restart kubelet
sudo systemctl daemon-reload
sudo systemctl restart kubelet
```

### Verifying the restore

```bash
kubectl get pods -n kube-system
kubectl describe pod etcd-master -n kube-system
kubectl get pods
kubectl get services
```

### Verifying restored data directory

```bash
sudo ls /var/lib/etcd
sudo ls /var/lib/etcd-restore-from-backup
```
