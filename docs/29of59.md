# Video 29 of 59 - Docker Storage (Prerequisite for Kubernetes Storage)

📺 https://www.youtube.com/watch?v=ZAPX21TMkkQ

## Overview

This video explains Docker storage concepts as a prerequisite for understanding Kubernetes storage (covered in the next video). It covers: layered architecture of Docker images, read-only layers vs. writable container layers, storage drivers vs. volume drivers, creating and using Docker volumes for persistent data, and bind mounts. The key takeaway is that container data is ephemeral unless you use volumes to persist it.


### Creating a Docker Volume

```bash
# Create a named volume
sudo docker volume create data_volume
```

### Inspecting Docker Storage Directories

```bash
# Default Docker storage location
sudo ls /var/lib/docker/

sudo ls /var/lib/docker/volumes/data_volume/_data
```

### Running a Container with Volume Mount

```bash
sudo docker run -v data_volume:/app -dp 3000:3000 --name todo day02-todo

```

### Testing Persistence with Volume

```bash
# Exec into the container
sudo docker exec -it <container-id> sh

# Navigate to the mounted path
cd /app

# Create a test file
echo "hello kubernetes" > test.txt
cat test.txt

exit
```

### Verifying Data Persists After Container Restart

```bash
# Stop the container
sudo docker stop <container-id>

# Check volume data still exists on host
sudo ls /var/lib/docker/volumes/data_volume/_data

# Restart the container
sudo docker start <container-id>

# Exec back in and verify
sudo docker exec -it <container-id> sh
cd /app
ls
# test.txt is still there
exit
```

### Verifying Data Persists After Container Removal

```bash
# Force remove the container
sudo docker rm -f <container-id>

# Check volume data still exists on host
sudo ls /var/lib/docker/volumes/data_volume/_data

# Run a new container with the same volume
sudo docker run -v data_volume:/app -dp 3000:3000 --name todo day02-todo

# Exec into the new container
sudo docker exec -it <container-id> sh
cd /app
ls
# test.txt is still there - data persisted!
exit
```

### Checking Docker Run Options

### Bind Mount Example

```bash
# Using --mount flag with bind type
sudo docker run --mount type=bind,source=/home/ubuntu/mydata,target=/app -dp 3000:3000 --name todo day02-todo

# Using -v flag (same effect)
sudo docker run -v /home/ubuntu/mydata:/app -dp 3000:3000 --name todo day02-todo
```

## Key Concepts

- **Storage Drivers**: Responsible for creating image layers and writing data to the container layer (overlay2, AUFS, ZFS, device-mapper)
- **Volume Drivers**: Responsible for making data persistent (local, AWS EBS, GCE Persistent Disk, Azure File Storage, Portworx, vSphere)
- **Volume Mount**: Maps a Docker-managed volume to a container path (`-v volume_name:/container/path`)
- **Bind Mount**: Maps a host filesystem path directly to a container path (`-v /host/path:/container/path` or `--mount type=bind,source=/host/path,target=/container/path`)
