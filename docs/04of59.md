# Docker Multi-Stage Builds

📺 [Watch on YouTube](https://www.youtube.com/watch?v=ajetvJmBvFo)

## Overview

Video 3 in the CKA series. Covers Docker multi-stage builds to reduce image size, improve performance, and follow best practices. Demonstrates building a React application with a two-stage Dockerfile: first stage installs dependencies and builds, second stage uses nginx to serve only the build artifacts. Result: smaller, more secure, production-ready images.

## Terminal Session: Clone and Setup

```bash
git clone <react-app-repo-url>
ls
cd <app-directory>
ls
```

## Terminal Session: Create the Multi-Stage Dockerfile

```bash
touch Dockerfile
vi Dockerfile
```

## Dockerfile (Multi-Stage Build)

```yaml
FROM node:18-alpine AS installer
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm install
COPY . .
RUN npm run build

FROM nginx:latest AS deployer
COPY --from=installer /app/build /usr/share/nginx/html
```

Save and exit: `Esc` → `:wq!`

## Terminal Session: Build the Multi-Stage Image

```bash
docker build -t multistage .
```

## Terminal Session: Verify Image Size

```bash
docker images
# multistage image: ~195 MB (smaller than the previous 217 MB, even with a larger app + nginx)
```

## Terminal Session: Cleanup Old Images

```bash
docker image rm <image-name-or-id>
```

## Terminal Session: Run the Container

```bash
docker ps
docker run -d -p 3000:3000 multistage
docker ps
```

## Terminal Session: View Container Logs

```bash
docker logs <container-id>
```

## Terminal Session: Exec into Container and Inspect

```bash
docker exec -it <container-id> sh

# Inside the container (default directory is / since no WORKDIR in second stage):
cd /usr/share/nginx/html
ls
# Shows: index.html and static files (only build artifacts, no node_modules)

exit
```

## Terminal Session: Inspect Container Details

```bash
docker inspect <container-id>
# Shows: MAC address, IPv4, sandbox key, host details, exposed ports, nginx version, etc.
```

## Key Takeaways

- Multi-stage builds use multiple `FROM` statements, each defining a "stage"
- Only the final stage becomes the actual image
- Use `COPY --from=<stage-name>` to copy artifacts from earlier stages
- Only copy what's needed for production (build artifacts, not node_modules)
- Benefits: smaller image size, better security, faster builds, production-ready
- Other best practices mentioned: run containers as non-root user
