# Video 10: Kubernetes Services (NodePort, ClusterIP, LoadBalancer)

📺 [Watch on YouTube](https://www.youtube.com/watch?v=tHAQWLKMTB0)

## Overview

This video covers Kubernetes Services: NodePort, ClusterIP, LoadBalancer, and ExternalName. It explains how services expose applications internally and externally, the difference between port, targetPort, and nodePort, and demonstrates creating each service type. The video also covers Kind cluster port mapping workaround, kubectl aliases, and bash completion.

## Key Concepts

- **NodePort**: Exposes service on a static port (range 30000-32767) on each node for external access
- **ClusterIP**: Internal-only service for pod-to-pod communication within the cluster (default type)
- **LoadBalancer**: Provisions an external load balancer (cloud provider) for external access
- **ExternalName**: Maps a service to an external DNS name
- **Port terminology**: `nodePort` (external) → `port` (service/internal) → `targetPort` (pod/container)

## YAML Manifests

### NodePort Service

```yaml
# nodeport.yaml
apiVersion: v1
kind: Service
metadata:
  name: nodeport-svc
  labels:
    env: demo
spec:
  type: NodePort
  ports:
    - nodePort: 30001
      port: 80
      targetPort: 80
  selector:
    env: demo
```

### ClusterIP Service

```yaml
# clusterip.yaml
apiVersion: v1
kind: Service
metadata:
  name: cluster-svc
  labels:
    env: demo
spec:
  ports:
    - port: 80
      targetPort: 80
  selector:
    env: demo
```

### LoadBalancer Service

```yaml
# lb.yaml
apiVersion: v1
kind: Service
metadata:
  name: lb-svc
  labels:
    env: demo
spec:
  type: LoadBalancer
  ports:
    - port: 80
      targetPort: 80
  selector:
    env: demo
```

### Kind Cluster Config with Port Mapping

```yaml
# kind.yaml - Kind cluster with port mapping for NodePort access
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
  - role: control-plane
    extraPortMappings:
      - containerPort: 30001
        hostPort: 30001
  - role: worker
  - role: worker
```

## Terminal Sessions

### Setting Up the Cluster (Kind-specific)

```bash
# Delete existing cluster
kind delete cluster --name cka-cluster-2

# Create new cluster with port mapping
kind create cluster --config kind.yaml --name cka-cluster-3

# Verify nodes
kubectl get nodes
```

### NodePort Service

```bash
# Check API version for service
kubectl explain service

# Create NodePort service
kubectl create -f nodeport.yaml

# Verify service
kubectl get svc

# Get pod details (IP, node)
kubectl get pods -o wide

# Get node IPs
kubectl get nodes -o wide

# Describe pod to find which node it's on
kubectl describe pod <pod-name>

# Test the service (on Kind cluster, use localhost)
curl localhost:30001

# On a regular cluster, use: curl <node-ip>:30001
```

### ClusterIP Service

```bash
# Show pod labels
kubectl get pods --show-labels

# Demonstrate pod IP changes on restart
kubectl describe pod <pod-name>  # note the IP
kubectl delete pod <pod-name>
kubectl get pods
kubectl describe pod <new-pod-name>  # IP has changed

# Create ClusterIP service
kubectl apply -f clusterip.yaml

# Verify
kubectl get svc

# Describe the service (shows endpoints)
kubectl describe svc cluster-svc

# Check endpoints
kubectl get endpoints
# or
kubectl get ep

# Verify pod IPs match endpoints
kubectl get pods -o wide
```

### LoadBalancer Service

```bash
# Create LoadBalancer service
kubectl create -f lb.yaml

# If service already exists, delete first
kubectl delete svc lb-svc
kubectl create -f lb.yaml

# Check services (external-ip will be <pending> without cloud LB)
kubectl get svc
```

### Exposing a Service Imperatively

```bash
# Expose a deployment as a service from command line
kubectl expose deployment nginx-deploy --port=80 --target-port=80 --type=NodePort
```

### Setting kubectl Alias and Bash Completion

```bash
# Add alias to bash profile
vi ~/.bash_profile
# Add: alias k='kubectl'

# Source the profile
source ~/.bash_profile

# Now use 'k' instead of 'kubectl'
k get pods
k get svc
k describe svc cluster-svc

# Bash completion (from cheat sheet)
source <(kubectl completion bash)
alias k=kubectl
complete -o default -F __start_kubectl k
```

### Describe and Troubleshoot Services

```bash
# Describe a service
kubectl describe service nodeport-svc

# Get all resources in the cluster
kubectl get all
```

## Key Points

- NodePort range: 30000–32767 (case-sensitive: `NodePort` with capital N and P)
- ClusterIP is the default service type (no `type` field needed)
- LoadBalancer requires a cloud provider or external LB provisioner
- ExternalName maps to an external DNS (e.g., `my.database.example.com`)
- Pod IPs are ephemeral — use services for stable access
- Service endpoints automatically update when pods restart with new IPs
- Kind clusters need `extraPortMappings` in config to expose NodePort externally
- On regular clusters/exam, NodePort is accessible at `<node-ip>:<nodePort>`
- Use `kubectl expose` for quick imperative service creation in the exam
